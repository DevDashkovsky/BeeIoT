# Этап 3: Тестировщик

## Роль
Senior QA Engineer и Code Reviewer для встраиваемых систем.

## Задача
Проверить реализованный код Разработчика на соответствие ТЗ, отказоустойчивость и корректность. Подготовить подробный отчёт с конкретными исправлениями.

## Входные данные
- Все файлы прошивки (выход Разработчика): `boot.py`, `main.py`, `config.py`, `sensors/*.py`, `network/sim800l.py`, `storage/ring_buffer.py`
- `docs/sensor_communication.md` — эталонные JSON-форматы
- `.agents/rules/task.md` — Мастер-промпт (критерии соответствия)

## Чеклист проверки

### Блок 1: Энергопотребление и Deep Sleep

Цель: гарантировать, что батарея не разрядится при любом сценарии.

- [ ] **Проверь все пути выполнения в `main.py`:**
  - Сценарий A: SIM800L не отвечает → вызывается ли `machine.deepsleep()`?
  - Сценарий B: GPRS недоступен → сохраняются ли данные в буфер → вызывается ли `deepsleep()`?
  - Сценарий C: MQTT-брокер недоступен → вызывается ли `deepsleep()`?
  - Сценарий D: Датчик завис или не ответил → возвращает ли он `-1` без зависания? → `deepsleep()` по расписанию?
  - Сценарий E: Необработанное исключение → ловится ли глобальным `try/except`? → `deepsleep()`?

- [ ] **Проверь `sim800l.power_off()`:**
  - Вызывается ли перед `deepsleep()` в КАЖДОМ из сценариев A–E?
  - Бросает ли `power_off()` исключение? (Не должна — должна быть обёрнута в `try/except` внутри)

Если найдены пути без `deepsleep()` — укажи конкретные строки кода и предложи исправление.

### Блок 2: Граничные случаи локального хранилища

Цель: кольцевой буфер должен работать корректно при переполнении.

- [ ] **Тест переполнения буфера:**
  - Если файл превышает `BUFFER_MAX_SIZE_BYTES` (10 КБ), удаляется ли самая СТАРАЯ запись?
  - Не удаляется ли несколько записей за раз нечаянно?
  - Корректно ли работает `_trim_to_limit` если файл содержит только одну строку?

- [ ] **Тест отсутствия файла:**
  - Что делает `pop_all()` если файл не существует? Не бросает ли исключение?
  - Что делает `is_empty()` если файл не существует?

- [ ] **Тест повреждённого JSON в буфере:**
  - Что если одна строка в JSONL-файле повреждена (не парсится)? Прерывается ли `pop_all()` или пропускает строку?

- [ ] **Тест создания директории:**
  - Создаётся ли `/data/` директория если она не существует?

Для каждой найденной проблемы — приведи конкретный фикс.

### Блок 3: Таймауты и отсутствие бесконечных циклов

Цель: устройство никогда не должно зависнуть навсегда.

- [ ] **Проверь все `while` циклы в коде:**
  - Есть ли у каждого цикла условие выхода по таймауту?
  - Примеры того что искать:
    - `while True: uart.read()` без `start_time = utime.ticks_ms()` и проверки `utime.ticks_diff()`
    - `while not registered: AT+CREG?` без `timeout`
  - Составь список ВСЕХ `while`-циклов в проекте и проверь каждый

- [ ] **GPRS timeout:**
  - Прерывается ли попытка подключения если нет ответа в течение 60 секунд (`GPRS_TIMEOUT_MS`)?

- [ ] **MQTT config wait:**
  - Ожидание топика `/config` не превышает 5 секунд (`MQTT_CONFIG_WAIT_MS`)?

---

### Блок 4: Корректность JSON-форматов

Цель: данные должны строго соответствовать спецификации из `docs/sensor_communication.md`.

- [ ] **Топик `/data`:** проверь что код формирует именно такой JSON:
  ```json
  {"temperature": float, "temperature_time": int, "noise": float, "noise_time": int}
  ```
  - Типы: `temperature`/`noise` — `float64`, `*_time` — `int64` (UNIX seconds)
  - При отсутствии данных значение `-1` (тип `int` или `-1.0` тип `float`?) — уточни по спецификации

- [ ] **Топик `/status`:** проверь:
  ```json
  {"battery_level": int, "signal_strength": int, "timestamp": int, "errors": []}
  ```
  - `errors` — всегда массив, даже если пустой
  - `battery_level` = -1 если нет данных (не `None`, не `null`)

- [ ] **Парсинг `/config`:** все поля обрабатываются правильно?
  - `sampling_rate_noise = -1` → не менять настройку (не перезаписывать в config)
  - `restart_device = true` → вызов `machine.reset()`?
  - `delete_device = true` → что происходит?

### Блок 5: Code Review — магические числа и чистота кода

Цель: все числа и строки — в `config.py`.

- [ ] **Поищи в коде (grep) следующие паттерны:**
  - IP-адреса или URL в виде строк вне `config.py`
  - Числовые таймауты (например `time.sleep(5)`) вне `config.py`
  - Имена файлов (`"/data/..."`) вне `config.py`
  - Пин-номера (например `Pin(4)`) хардкоженные вне `config.py`
  - APN строки вне `config.py`

  Для каждого найденного — укажи файл, строку и предложи исправление.

- [ ] **Проверь flash-память:**
  - В `ring_buffer.py` не используется ли `open(file, 'r').read()` для больших файлов без частичного чтения? (На ESP32 ОЗУ ограничено!)
  - Оптимальна ли логика `_trim_to_limit`? Не деградирует ли производительность с ростом файла?
